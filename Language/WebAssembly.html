WebAssembly

  • WebAssembly 入门课
  • WebAssembly的长久目标
  • WebAssembly使用场景

特性

  • 本质是中间字节码。 (WebAssembly 到底处于编译阶段的哪个环节？)
  • 计算模型是堆栈机模型 (Stack Machine，很多VM指令集都是基于该模型)
  • 一次编译到处运行，具有跨平台特性。 (可以类比为Java字节码与JVM)
  • 相比JS能够运行更快。(省略编译步骤，运行时也不需要再实时编译。)
  • JS与WebAssembly可以互操作。(C++中调用JS: Calling JavaScript code from C/C++ using WebAssembly)
  • 代码安全性高，需要通过逆向来分析代码逻辑
  • 为了保证JS与WASM的互操作性，基本JS引擎都直接支持了WASM的执行(如V8内可以执行WASM)。JS引擎要支持WASM，需要实现相应规范。
  • WASM是字节码二进制形式，WAT是字节码可读形式。两者是等价的。(wabt，转换工具集)
  • 可以利用LLVM对中间码进行优化。(即会先生成LLVM IR，进行相应优化，再转为WASM) (C++场景下LLVM IR与WebAssembly的关系)
  • LLVM后端，可以生成WebAssembly目标。(LLVM GettingStarted)

WASM快的原因

JavaScript是边解释边执行，即使支持JIT，也只针对部分代码进行优化。具体JS执行过程可以查看：JS执行过程详解

WASM，最简单的理解就是跑在JVM上的中间字节码，本身不需要于走语法分析、词法分析等阶段。

Emscripten

基于LLVM/Clang的编译器，它将C/C++源码编译为WebAssembly。编译出来的代码可以在浏览器、Node.js等中执行。

Emscripten也是支持将C/C++编译为asm.js的。

asm.js

asm.js 和 Emscripten 入门教程

asm.js可以理解为wasm的前身，也是通过符合指定的条件，来让JS代码在某些JS引擎上执行更快。asm.js更多是面向JS引擎，而非普通开发者的。

  • 它的变量一律都是静态类型
  • 取消垃圾回收机制
  • 是JS的一个严格子集 （即不支持asm.js的引擎也可以正常运行）
  • JavaScript引擎识别出asm.js，就跳过语法分析，直接转成字节码
  • 速度大约是原生代码的50%
  • 不是面向开发者，而是从其它语言编译而来
