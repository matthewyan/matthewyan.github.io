Debugger原理：Debugger 原理揭秘、调试程序时，设置断点的原理是什么？

Debugger插入断点，就是在机器码中相应位置替换为0xcc(即INT 3)，也即操作系统软中断，执行到该位置自然会断下来。断下下之后会立即将该位置的指令码还原，以便于继续调试执行。 其实还有硬中断，也可以实现断点。

LLDB能力：

  • 断点相关操作
  • 控制执行流程
  • 执行某些代码
  • ……

命令

  • expression somesth：执行某个表达式

  • call somesth：调用某个方法

  • bt 5：打印当前堆栈，且只打印5帧

  • thread list：查看线程列表

  • thread return 10：直接从当前方法中返回，且返回值为10

  • c：继续

  • n：下一步

  • s：进入

  • f：跳出

  • frame select N：当前堆栈中，跳至第N帧

  • frame variable ***：查看帧变量（不指定具体变量，查看所有本地变量）

  • target variable ***：查看全局变量（不指定具体变量名，即可查看所有）

  • image lookup -address 0x***：查看指定地址的信息

  • image lookup -name：查找方法来源，可以查看所有库中指定名称的方法

  • image lookup -type：查看某个类的所有成员

  • breakpoint set -f ViewController.m -l 30：在指定文件中指定行打断点

  • breakpoint set -n viewDidLoad：会在所有viewDidLoad上打断点

  • breakpoint set -n "-[MobShareViewController viewDidLoad]"：在指定类的指定方法上打断点

  • breakpoint delete 编号：删除指定断点（不指定编号即清除所有）

  • breakpoint set -f MOBQQViewController.m -l 50 -c "parameters != nil"：在指定位置添加符号断点，判断条件是c参数之后的字符串

  • breakpoint list：查看断点列表

  • breakpoint disable/enable 2：禁用/启用指定断点

  • watchpoint set variable b：设置观察点，监听b变量，当其有改变时，会断下来

  • watchpoint set expression 0x7ffee1c1f0fc：直接对指定内存地址进行监听

  • watchpoint list：查看观察点

  • watchpoint delete 编号：删除观察点（不指定编号即清除所有）

  • target stop-hook command：可以在断点断下来之后，就自动执行指定的command

  • target stop-hook list：列出所有的hook指令

  • memory find -s "str" addr1 addr2: 从指定的内存地址区间之中，搜索指定的字符串

  • image list: 查看所有库的列表以及加载的内存地址

观察点，有软件/硬件实现之分。软件实现的话，执行速度比正常执行要慢非常多（上百倍）。硬件观察点，速度基本不受影响。目前Xcode调试也支持了硬件观察点，设置观察点时就可以看到相关信息：

动态库 & 静态库

  • 动态库的加载顺序
  • iOS 开发中的『库』

静态库：编译时跟App链接在一起，启动时间会更少、安装包大小更小。

动态库：动态库分两种，一种是系统的tbd、dylib。一种是.framework。

  • tbd是系统使用的，真正的动态库。
  • framework是动态加载到进程内存空间的，更多是为了同一进程内的多个目标使用。比如宿主和Extension可以共享。
  • framework可以使用dlopen来加载。
  • .tbd 其实是一个YAML本文文件，描述了需要链接的动态库的信息。主要目的是为了减少app 的下载大小。

苹果在Xcode11中支持了一种新的库类型：XCFramework，可以将多个平台的二进制打包在一起。(即iOS/Mac的库可以打包在一起) 具体参考：XCFramework 是用来干什么的

ModuleMap

理解 iOS 中的 Modules

  • ModuleMap是将多个module (可以理解为一个.o) 合并为一个module
  • 可以用来作为OC / Swift的桥接手段 (其实可以扩展至其它语言)
  • 可以替换传传统的OC/C的头文件导入方式，直接 import 导入
  • C/C++头文件，被include几次，就会编译几次，而Module方式只需编译一次
  • 框架开了Defines Module后，普通的 #include和#import都会被替换为@import
  • ModuleMap有一个统一的头文件，并且会被编译为二进制形式的

崩溃

iOS 崩溃千奇百怪，如何全面监控？

  • Crash Sdk是通过注册异常信号来捕获异常的。
  • 捕捉到异常再用backtrace接口来获取当前堆栈信息

崩溃捕捉常见流程：

  • 使用NSGetUncaughtExceptionHandler()保存现有的Crash异常处理方法
  • 使用signal来捕获指定信号，有些信号代表异常。并设置处理方法 (如signal(SIGILL, signalHandler))
  • 使用NSSetUncaughtExceptionHandler获取OC异常，设置异常捕获处理方法
  • 处理完之后，需要调用一下开始保存的处理方法

原生渲染

iOS原生、大前端和Flutter分别是怎么渲染的？

  • 更新视图树，同步更新图层树
  • CPU计算要显示的内容（包括布局计算、图像解码等），然后转至Render Server
  • 数据到达Render Server后，按照图层树中图层顺序、RGBA值等等，转成渲染树给OpenGL / Metal
  • Render Server调用GPU进行处理

前三步，被称为Commit Transaction

在页面静止时，FPS一般是0，只有需要更新时，页面才需要刷新。(关于iOS性能优化)

How to take a screenshot programmatically in iPhone/iOS?

示例代码：

func takeshot(_ sender: Any) {
   var image :UIImage?
   let currentLayer = UIApplication.shared.keyWindow!.layer
   let currentScale = UIScreen.main.scale
   UIGraphicsBeginImageContextWithOptions(currentLayer.frame.size, false, currentScale);
   guard let currentContext = UIGraphicsGetCurrentContext() else {return}
   currentLayer.render(in: currentContext)
   image = UIGraphicsGetImageFromCurrentImageContext()
   UIGraphicsEndImageContext()
   guard let img = image else { return }
   UIImageWriteToSavedPhotosAlbum(img, nil, nil, nil)
}

Passkey

遇见 Passkey

PassKey是苹果对WebAuthn的实现，其核心验证思路就是：

  • 服务端(需要支持WebAuthn)下发challenge，给浏览器
  • 浏览器再转给认证器(如指纹认证模块)，认证器用自己的私钥加密，再通过浏览器传回服务端
  • 服务端使用公钥进行解密，如果能正常解出来，说明相应的端是可信的

Private Access Token (PAT)

验证码的替代者—私有访问凭证
