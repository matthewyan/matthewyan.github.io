React Native

最简单的理解，React Native类似NodeJS，都是通过扩展JS引擎，使JS代码有了调用Native接口的能力。

RN目前依赖的JS引擎是JavaScriptCore，是Webkit中的JS引擎。在iOS上是自带的，但在安卓上，需要打包进应用，所以RN的安卓版本要更大。不过在新的RN版本后，有支持Hermes引擎，Hermes是FaceBook专门针对RN安卓平台优化的引擎。

核心概念

RN Bundle

简述 ReactNative Bundle

RN Bundle本质就是js包，为什么需要RN Bundle呢？

  • 将React(JSX)转为JS
  • 支持转为ES5
  • 依赖外部JS模块
  • 代码混淆

RN Bundle组成部分：

  • polyfills (做一些JS Context中的准备工作，如定义模块方法等)
  • module 定义 (业务逻辑)
  • require 调用 (用来加载必须的初始化及根组件)

Bundle有两种类型： (分析一下metro下两种Bundle格式)

  • Plain Bundle
  • RAM Bundle
Module ID

是从0开始递增的数字，用来唯一标识各个Module

Module Factory

作用：创建并导出组件类，并注册根组件。

原理 & 源码

RN使用及工作原理，简单来说，有如下几点：

  • 开发层面，非常类似ReactJS，包括文档也主要参考ReactJS

  • RN中也有Virtual DOM的概念。布局变化也会通过Diff算出差异，提升布局效率

  • JS中的布局逻辑，会转换为操作指令，调用至Native侧

  • JS和Native的交互非常频繁，但效率不高，这儿也是RN效率较低的根本原因

  • 在新版本的RN(0.59之后)中，提供了JSI来提升JS和Native的交互效率

  • Native调用JS，是通过JSC提供的JSObjectCallAsFunction等接口实现的

  • JS调用Native，是通过JSC提供的JSContext的block能力。使用示例：jsContext[@"funcname"] = ^() {}

新版本的RN的OC和JS的交互已经通过JSI来实现了

在iOS中，JSI并不是简单使用的JSC的evaluateScript:方法执行JS，而是使用了JSC提供的原始C接口。具体可以参考RN源码：JSCRuntime::evaluateJavaScript的实现。（Pod中的React-jsi即是JSI的实现）

JS调用OC的方法，使用的是JSC提供的block方式。在新版本中，是在C++中实现的，参考源码：JSIExecutor::initializeRuntime()的实现。

JavaScriptCore 提供的C接口可以参考：JavaScriptCoreC语言API详细介绍

RN中Native和JS交互的逻辑，说的比较清楚是这篇文章：RN 通信原理（for 前端）。 对应的示例代码为：iostalks/Arch

渲染原理

React Native 渲染原理

  • RN也会维护一个Virtual DOM，在内存中的一个树形JS Object

  • RN通过Bridge将需要变更的指令，传递给Native (Web上，直接通过DOM API来操作真实的DOM)

  • 在Native侧，维护了一个Shadow Tree，与Virtual DOM对应，

Native调用JS

ReactNative IOS 运行原理之

  • 外部调用接口：RCTBridge enqueueJSCall:(NSString *)moduleDotMethod args:(NSArray *)args
  • 核心执行接口：NativeToJsBridge::callFunction
  • 支持OC调用JS前，需要先绑定Native&JS，相应逻辑在JSIExecutor::bindBridge中处理

执行顺序：

  • RCTMessageThread::runOnQueue 在JSThread中执行指定block （或NativeToJsBridge::runOnExecutorQueue）
  • tryAndReturnError 尝试执行相应逻辑，出了异常就会被捕获，并生成：Exception *** was thrown from JS thread
  • NativeToJsBridge::callFunction
  • JSIExecutor::callFunction

RCTRootView

RN在Native上的展示容器，所以RN基于这种实现，就很方便在已有的Native应用中进行插入某一个RN页面

RCTShadowView

继承自NSObject，主要能力是在子线程进行布局相关的计算。与RCTView的关系类似UIView和CALayer的关系。

RCTBridge

负责管理OC和JS的交互。 所有要暴露给JS的Native模块，都需要实现RCTBridgeModule协议

特点：

  • Fabric架构，直接生成的C++版本的渲染树，无须从JS再到Native转换
  • JSI不保证线程安全，多个线程访问同个HostObject时，就可能会出现数据竞争的情况

原理

  • RN通信原理 -- JavaScriptCore的应用

TurboModule之所以能工作，在于JSC(或Hermes)提供了JSObjectMake接口。这个接口的声明：JSObjectRef JSObjectMake(JSContextRef ctx, JSClassRef jsClass, void* data);

也即可以将一个JSClass绑定到C++对象。但JSClassRef需要提前声明，并明确告知JSEngine，具体的接口实现都可以放在C++侧。

Fabric工作流程

（来源于ChatGPT，待确认）

  • 初始化：App 启动后，React Native Fabric 会初始化一个 Fabric 队列，并从此队列中获取任务。
  • 根节点渲染：React Native Fabric 会在 Fabric 队列中找到根节点任务，然后使用 Flexbox 布局算法计算每个子节点的位置。
  • 生成布局树：React Native Fabric 会根据计算出的位置信息生成布局树，这个树包含了所有子节点的位置和布局信息。
  • 生成渲染树：React Native Fabric 将布局树转换成渲染树，渲染树包含了所有需要被渲染的节点信息，如文、图片等。
  • 绘制：React Native Fabric 会使用 Skia 图形引擎将渲染树转换成屏幕上的像素，然后再将像素渲染到屏幕上。
  • 布局更新：当组件的状态发生变化时，React Native Fabric 会重新计算布局，并生成新的布局树和渲染树。
  • 安排渲染：React Native Fabric 会将需要重新渲染的节点添加到 Fabric 队列中，并在下一次渲染周期中重新渲染这些节点。
  • 垃圾回收：当一个节点不再需要被渲染时，React Native Fabric 会将它从渲染树中删除，并释放它占用的内存。
